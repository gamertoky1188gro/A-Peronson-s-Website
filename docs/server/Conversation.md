# Conversation - Server Feature Documentation (Manual)

## File Structure & Overview
- `server/routes/conversationRoutes.js`: Conversation lock claim/grant routes.
- `server/controllers/conversationController.js`: Claim/grant response mapping.
- `server/services/conversationLockService.js`: Lock ownership, access grant, and notification emission.
- `server/database/conversation_locks.json`: Lock state per request.
- `server/database/notifications.json`: Notifications generated by claim/grant operations.

## Code Explanation

### `conversationRoutes.js`
- Routes:
  - `POST /api/conversations/:requestId/claim`
  - `POST /api/conversations/:requestId/grant`
- Both require auth and roles `buying_house` or `admin`.

### `conversationController.js`
- `claim(req, res)`:
  - calls `claimConversation(requestId, req.user)`.
  - returns `409` when service status is `locked`.
  - otherwise returns service payload.
- `grant(req, res)`:
  - calls `grantConversationAccess(requestId, ownerId, target_agent_id)`.
  - maps service sentinel values:
    - `invalid_target` -> `400`
    - `forbidden` -> `403`
    - `null` -> `404`
  - success -> `200`.

### `conversationLockService.js`
Functions:
- `claimConversation(requestId, agent)`:
  - if no lock exists:
    - creates lock with `locked_by=agent.id` and `allowed_agents=[agent.id]`.
    - writes `conversation_locks.json`.
    - writes self-notification.
    - returns `{status:'claimed', ...lock}`.
  - if existing lock belongs to or already includes agent:
    - returns `{status:'granted', ...lock}`.
  - else:
    - returns `{status:'locked', notification:'...'}`.
- `grantConversationAccess(requestId, ownerId, targetAgentId)`:
  - validates target id.
  - checks lock exists and requester is lock owner.
  - appends target agent into `allowed_agents`.
  - writes grant notification to target.
  - returns updated lock.

Auxiliary:
- `createLockNotification` writes `conversation_lock` notification records.

## API Endpoints
- `POST /api/conversations/:requestId/claim`
  - Auth + role required.
  - Body: none.
  - Responses:
    - `200`: claimed/granted lock object
    - `409`: lock owned by another agent
- `POST /api/conversations/:requestId/grant`
  - Auth + role required.
  - Body:
```json
{ "target_agent_id": "user-123" }
```
  - Responses:
    - `200`: updated lock object
    - `400`, `403`, `404`

## Database / Data Model
- `conversation_locks.json` row:
  - `request_id`
  - `locked_by`
  - `allowed_agents: string[]`
  - `created_at`
- `notifications.json` row (conversation lock type):
  - `type: 'conversation_lock'`
  - `entity_type: 'buyer_request'`
  - `entity_id: request_id`
  - `actor_id`
  - `message`
  - `read`
  - timestamps

## Business Logic & Workflow
1. Buying-house/admin agent claims buyer request conversation.
2. First claimer becomes lock owner.
3. Other agents receive locked response unless explicitly granted.
4. Lock owner grants access to selected agent IDs.
5. Grant action emits notification to recipient.

## Error Handling & Validation
- Conflict (`409`) used for claimed-by-other scenario.
- Missing target agent id on grant -> `400`.
- Owner mismatch on grant -> `403`.
- Missing lock on grant -> `404`.

## Security Considerations
- Auth and role gating at route level.
- Ownership enforcement before grant mutation.
- Notifications scoped to specific user IDs.

## Extra Notes / Metadata
- Locking model is request-level, not match-thread-level; request/thread id linkage should remain consistent across modules.
